# MissAV Manager v0.1 使用教程

本文档面向当前桌面 App 版本 v0.1，目标是把“从原始文本/Telegram HTML 提取番号，到本地库整理，再到导出 Raindrop 文件”的完整流程讲清楚。

## 1. 软件定位

MissAV Manager 是一个本地桌面工具，主要做四件事：

1. 从任意文本、HTML、TXT、MD、CSV、LOG 中提取番号。
2. 为番号生成 MissAV 候选链接并尝试访问，抓取女优 tag 和类型 tag。
3. 把结果写入本地 SQLite 数据库，方便后续编辑、核验、备份和导出。
4. 导出 Raindrop 可导入的 HTML / CSV 文件。

它不是在线账号系统，数据主要保存在本机；你可以把它理解成“番号整理数据库 + Raindrop 导入生成器”。

## 2. 启动方式

### 2.1 直接运行便携版（推荐给普通用户）

从 GitHub 仓库的 Releases 页面下载 `MissAV_Manager_v*.exe`，保存到任意可写文件夹后双击即可启动。

- 不需要安装 Node.js、Git 或 npm。
- EXE 可以移动位置；数据库不会跟着 EXE 移动，也不会因升级 EXE 被覆盖。
- 这是未签名的个人桌面软件，Windows SmartScreen 可能显示提示。只应从项目 Releases 下载。

### 2.2 从源码启动（开发者）

在当前文件夹中可以直接使用：

```bash
npm start
```

如果需要开发模式和调试窗口：

```bash
npm run dev
```

也可以双击项目内的：

```text
启动MissAV.bat
```

### 2.3 首次准备

如果依赖不存在，先执行：

```bash
npm install
```

### 2.4 构建便携版

```bash
npm run build:portable
```

构建结果会输出到 `dist/MissAV_Manager_v<版本号>.exe`。这个 EXE 适合上传到 GitHub Release，供其他 Windows 用户直接下载。

## 3. 数据保存在哪里

App 使用 SQLite 本地数据库保存主要数据。界面中进入：

```text
外观 -> 数据位置 -> 数据库目录
```

可以看到数据库所在目录。

便携版会把数据保存到当前 Windows 用户的应用数据目录，而不是 EXE 所在目录。这样升级或替换 EXE 时不会覆盖已有数据库；也意味着每个 Windows 用户都有自己的独立本地库。

常见数据类型：

| 数据 | 保存位置 / 来源 |
| --- | --- |
| 收藏记录 | SQLite 本地库的 `bookmarks` 表，完整保存 Raindrop 字段 |
| 历史番号索引 | SQLite 本地库的 `codes` 表，用于全量去重 |
| 女优 tag | SQLite 本地库的 `actress_tags` 表 |
| 类型 tag | SQLite 本地库的 `genre_tags` 表 |
| 番号与女优关系 | `actress_code_map` 表 |
| 番号与类型关系 | `code_genres` 表 |
| 处理批次 | `processing_runs` 表 |
| CSV 临时编辑 | CSV 工作台打开的外部文件 |
| 外观主题 | 本地设置缓存 |

重要原则：

- SQLite 本地库是主数据库。
- CSV 工作台既能无损编辑 Raindrop 官方 CSV，也兼容旧女优 Tag CSV。
- 做大批量修改前，建议先在“备份恢复”里创建备份。

## 4. 推荐使用流程

最稳妥的工作流如下：

```text
导入/粘贴原始文本 -> 提取番号 -> 开始处理 -> 查看结果 -> 进入本地库整理 -> 整理卡片/导出预览 -> 导出 Raindrop CSV 或 HTML
```

如果你只是维护已有数据，可以直接从“数据 -> SQLite 本地库”开始。

## 6. 番号处理区

位置：

```text
处理 -> 番号处理
```

### 6.1 支持输入什么

可以直接粘贴：

- 普通番号列表，例如 `DASS-720`、`MIDV-682`。
- 无横杠但大写的番号，例如 `SONE229`。
- FC2，例如 `FC2-PPV-4843473`、`FC2 4843473`。
- MissAV 链接，例如 `https://missav.ai/dm89/cn/DASS-720`。
- Telegram 导出的 HTML 片段。
- Markdown、TXT、CSV、LOG、JSON 等纯文本内容。

也可以点击：

| 按钮 | 用途 |
| --- | --- |
| 导入文件 | 读取单个 txt/html/htm/md/json/csv/log 文件，并当作纯文本提取番号 |
| 批量导入 | 一次读取多个文件，合并文本后提取番号 |
| 复制识别结果 | 把当前识别出的番号复制到剪贴板 |
| 保存清单 | 保存当前识别出的番号列表为 txt |
| 清空 | 清空输入框 |
| 示例 | 填入内置示例数据 |

### 6.2 提取逻辑

当前提取规则是：

1. 优先识别 MissAV URL，从 URL slug 中提取番号。
2. 识别 FC2 系列并标准化为 `FC2-PPV-数字`。
3. 识别带分隔符的普通番号，例如 `ABF-354`、`ABF 354`、`ABF_354`。
4. 识别无分隔符但原文大写的普通番号，例如 `SONE229`。
5. 自动去重，保留首次出现顺序。

### 6.3 噪声过滤逻辑

为了避免 Telegram HTML 里的 class/id 被误判，App 会过滤明显噪声前缀，例如：

```text
MESSAGE, MESSAGES, USERPIC, MEDIA, VIDEO, PHOTO, PAGINATION, DETAILS,
STATUS, TITLE, BODY, CLASS, STYLE, DATE, HTML, BUTTON, INPUT, IMAGE,
THUMB, AV, TOP, BEST, MOODYZ, TAMEIKE
```

例如：

| 原始文本 | 处理结果 |
| --- | --- |
| `message14298` | 不识别为番号 |
| `MESSAGE-13` | 过滤 |
| `moodyz` | 过滤 |
| `DASS-720` | 识别 |
| `https://missav.ai/dm89/cn/DASS-720` | 识别为 `DASS-720` |

注意：无分隔符识别只接受原文大写前缀，这是为了避免 `message14298` 这类 HTML 噪声。

## 7. 开始处理

位置：

```text
处理 -> 运行控制
```

步骤：

1. 在“番号处理”中粘贴或导入文本。
2. 确认右上角识别数量正确。
3. 在“输出目录”点击“选择目录”。
4. 点击“开始处理”。
5. 处理过程中可以点击“停止处理”。

### 7.1 处理每个番号时会做什么

对每个番号，App 会：

1. 标准化番号。
2. 生成多个候选 MissAV 链接。
3. 逐个尝试访问候选链接。
4. 判断页面是否包含目标番号。
5. 从页面中提取女优 tag 和类型 tag。
6. 写入 SQLite 本地库。
7. 在结果页显示状态、链接和 tags。

普通番号候选链接大致包括：

```text
https://missav.ai/cn/{lower-code}
https://missav.ai/cn/{lower-code}-chinese-subtitle
https://missav.ai/cn/{lower-code}-uncensored-leak
https://missav.ai/dm96/cn/{CODE}
https://missav.ai/dm89/cn/{CODE}
```

FC2 会生成 FC2 专用候选链接。

### 7.2 状态含义

| 状态 | 含义 | 建议 |
| --- | --- | --- |
| `ok` | 页面可访问，且能提取到有效信息 | 可直接整理/导出 |
| `page_ok_play_unknown` | 页面能打开，有女优信息，但播放状态不完全确定 | 建议抽查 |
| `no_actress_found` | 页面能打开，但没有提取到女优 tag | 补充女优 tag 或标记 `#未知女优` |
| `need_manual_check` | 链接可能可访问，但页面内容不足或无法可靠判断 | 人工打开核验 |
| `not_found` | 候选链接均未确认有效 | 进入人工查找流程 |
| `already_exists` | 本地库已有该番号 | 默认跳过本次处理 |
| `network_error` | 网络或请求失败 | 稍后重试或手动核验 |

重要：如果一条记录“链接可以打开”，但状态仍是 `not_found`，导出时可能被当成“需要查找”。这时请在“整理卡片”或“收藏库”里把状态改为 `ok` 或 `need_manual_check`，并确认 tags。

## 9. 输出文件说明

每次导出会创建类似这样的文件夹：

```text
YYYYMMDD_HHMM_missav_import/
```

里面通常包括：

| 文件 | 用途 |
| --- | --- |
| `*_missav_raindrop_import.html` | Raindrop 可导入 HTML |
| `*_missav_raindrop_import.csv` | Raindrop 可导入 CSV |
| `*_missav_import_report.csv` | 本次处理报告 |
| `*_女优tag合集.csv` | 从本地库导出的女优 tag 合集 |
| `*_missav_backup.json` | 本次结果备份 |

Raindrop 官方 CSV 字段为：

```text
id,title,note,excerpt,url,folder,tags,created,cover,highlights,favorite
```

CSV 包含 Raindrop ID 和 Excerpt，是优先使用的无损备份格式；HTML 可补充 Collection 层级、Last Modified、Notes 和 Highlights。导入同一批备份时可以一次同时选择 CSV 和 HTML。

本地库是 App 的核心数据区。顶部有搜索框，可以搜索：

本地库页签包括：

| 导出预览 | 导出前查看 CSV 预览和问题提醒 |
| 数据体检 | 检查无链接、缺 tag、状态矛盾、孤立 tag、坏关联 |
| 备份恢复 | 创建、恢复、删除数据库备份 |

## 11. 女优 Tag 页

位置：

```text
数据 -> SQLite 本地库 -> 女优 Tag
```

可执行操作：

| 操作 | 说明 |
| --- | --- |
| 新增女优 Tag | 手动创建一个 tag |
| 重命名 | 修改 tag 名称 |
| 合并 | 把一个 tag 的关联番号合并到另一个 tag，并删除源 tag |
| 删除 | 删除 tag，同时移除它和番号的关联 |

适合处理：

- 同一女优出现多个别名。
- 错别字 tag。
- 没有关联番号的无效 tag。

## 12. 收藏库页

位置：

```text
数据 -> SQLite 本地库 -> 收藏库
```

这是最重要的编辑页面，布局为 Collection、记录列表和详情编辑器。普通网页收藏不要求存在番号；能识别出番号的记录会自动关联到历史去重索引。

Collection 是真正的层级目录：`日本av / MissAV_Saved_Actress_Tags` 会按两级显示。父文件夹会筛选本身和所有子文件夹的记录；点击顶部“+ 新建”使用应用内弹窗创建 Collection，右键文件夹或点击行末 `⋮` 可新建子文件夹、重命名、选择范围或删除。右键“全部收藏”可清空全部收藏和实际文件夹，右键“未分类”可清空全部未分类记录；这两个入口是系统视图，不会从侧栏消失。删除操作不会删除历史番号去重索引。

### 12.1 Collection 与记录列表

列表支持：

| 按钮 | 用途 |
| --- | --- |
| 点击记录 | 在右侧打开完整编辑面板 |
| 星标 | 直接切换 Favorite |
| 打开 | 用浏览器打开链接 |
| 删除 | 删除收藏；历史番号索引继续保留用于去重 |

### 12.2 右侧 Raindrop 编辑面板

可编辑字段：

| 字段 | 导出到 Raindrop 的含义 |
| --- | --- |
| Link | `url` |
| Title | `title` |
| Collection | `folder` |
| Tags | `tags` |
| Note | `note` |
| Created | `created` |
| Excerpt | 备用摘要，可用于生成 note |
| Cover | 封面 URL，用于本地预览/记录 |
| Highlights | Raindrop 高亮内容 |
| Favorite | Raindrop 收藏标记 |
| Raindrop ID | 官方 CSV 中的稳定记录 ID |
| Last Modified | HTML 备份中的最后修改时间 |

番号索引区域还可编辑：

| 字段 | 说明 |
| --- | --- |
| 番号 | 数据库中的 code |
| 状态 | 数据库中的 status |
| 女优 Tag | 一行一个或用逗号/顿号/分号分隔 |
| 类型 Tag | 一行一个或用逗号/顿号/分号分隔 |

修改后点击“保存”或“保存并下一条”；也可按 `Ctrl+S` 保存。面板出现“未保存”提示时，说明当前改动还没有写入数据库。清空番号只解除当前收藏的索引关联，不会删除历史去重记录。

### 12.3 批量编辑

收藏库顶部支持：

| 操作 | 用途 |
| --- | --- |
| 选择当前列表 | 勾选当前筛选结果 |
| 清空选择 | 取消选择 |
| 复制番号 | 复制选中番号 |
| 导出 Raindrop CSV / HTML | 导出当前本地库中的 Raindrop 文件 |
| 批量状态 | 对选中记录应用状态 |
| 追加/移除女优 | 批量修改女优 tag |
| 追加/移除类型 | 批量修改类型 tag |
| 生成链接 | 为选中记录生成候选链接 |
| 规范番号 | 把番号改成标准格式 |
| 删除选中 | 批量删除 |
| 选择当前范围 | 一次选中当前搜索、筛选和 Collection 子树范围内的全部记录，不受分页限制 |

Raindrop 批量编辑区支持：

| 操作 | 用途 |
| --- | --- |
| 应用 Collection | 批量设置 Collection |
| Tags 追加/替换/删除 | 批量处理 Raindrop tags |

批量编辑前建议先在“备份恢复”创建备份。部分批量操作会自动创建备份，但手动备份更稳。

常用按钮：

## 14. 整理卡片

位置：

```text
数据 -> SQLite 本地库 -> 整理卡片
```

整理卡片适合逐条修正数据，不用在大表格里来回找。

### 14.1 队列类型

| 队列 | 包含内容 |
| --- | --- |
| 优先处理 | 无链接、状态矛盾、缺核心 tag 等较重要问题 |
| 链接问题 | 没有链接，导出时会被排除 |
| 状态核验 | 有链接但状态异常，例如 `not_found`、`need_manual_check` |
| 元数据整理 | 缺 Title、Collection、Tags、Created |
| 备注封面 | 缺 Note、Excerpt、Cover |
| 全部记录 | 全库记录 |

### 14.2 单卡字段

每张卡片分两部分：

| 区域 | 内容 |
| --- | --- |
| Raindrop | Link、Title、Collection、Tags、Note、Created |
| 本地库 | 番号、状态、女优 Tag、类型 Tag、Metadata |

### 14.3 快速修复按钮

| 按钮 | 作用 |
| --- | --- |
| 生成链接 | 根据番号生成 MissAV 候选链接 |
| 状态 ok | 把状态改为 `ok` |
| 标题=番号 | 把 Title 设置为当前番号 |
| 默认 Collection | `ok` 记录设为 `MissAV_Import`，`not_found` 设为 `需要手动核验` |
| 自动 Tags | 用女优 tag + 类型 tag 生成 Raindrop Tags |
| Created=现在 | 设置当前时间 |
| 生成备注 | 用番号、女优、类型生成一行 Note |
| 清空封面 | 清空 Cover 字段 |

### 14.4 保存与跳转

| 按钮 | 作用 |
| --- | --- |
| 保存并下一条 | 写入数据库并进入下一张卡 |
| 仅保存 | 写入数据库，停留在当前卡 |
| 跳过 | 不保存当前修改，进入下一张卡 |
| 撤销 | 放弃当前未保存修改并刷新 |
| 删除 | 删除当前番号 |

实用建议：

- 先处理“链接问题”和“状态核验”。
- 对可访问但状态是 `not_found` 的记录，改成 `ok` 或 `need_manual_check`。
- 对缺 tag 的记录，先补女优/类型，再点“自动 Tags”。
- 处理完一批后去“导出预览”检查。

## 15. 导出预览

位置：

```text
数据 -> SQLite 本地库 -> 导出预览
```

导出预览用于检查即将生成的 Raindrop 文件。

会显示：

| 区域 | 说明 |
| --- | --- |
| 本次扫描 | 当前搜索条件下的总记录 |
| 可导出 | 有有效链接的记录 |
| 无链接排除 | 不会进入导入文件的记录 |
| 未找到状态 | 需要人工确认的记录 |
| 提醒项 | 默认值或缺失字段 |
| CSV 预览 | 最终导出的 CSV 前若干行 |
| 问题与提醒 | 点击可定位到收藏库编辑 |

导出前建议至少确认：

- 无链接排除数量是否符合预期。
- 有链接但 `not_found` 的记录是否已修正。
- Tags 是否为空。
- Collection 是否是你想要的。

## 16. 数据体检

位置：

```text
数据 -> SQLite 本地库 -> 数据体检
```

数据体检会检查：

| 项目 | 说明 |
| --- | --- |
| 无链接 | `best_url` 为空 |
| 无女优 | 没有关联女优 tag |
| 无类型 | 没有关联类型 tag |
| 需人工核验 | 状态为 `need_manual_check`、`no_actress_found`、`page_ok_play_unknown`、`network_error` |
| 未找到 | 状态为 `not_found` |
| 疑似重复组 | 去横杠后相同或 FC2 变体相同 |
| 孤立女优 | 没有关联番号的女优 tag |
| 孤立类型 | 没有关联番号的类型 tag |
| 坏关联 | 关联指向不存在的番号或 tag |
| 状态矛盾 | 例如 `ok` 但无链接，或 `not_found` 但有链接 |

常用修复：

| 按钮 | 用途 |
| --- | --- |
| 为无链接生成链接 | 给体检样本中的无链接记录生成候选链接 |
| 清理孤立 Tag | 删除没有关联的 tag |
| 清理坏关联 | 删除无效关联行 |
| 查看无链接 / 无女优 / 需核验 | 跳转到收藏库对应筛选 |

## 20. 疑似重复


## 24. Raindrop 导入建议

推荐迁移和导出顺序：

1. 从 Raindrop 同时导出 CSV 和 HTML 备份。
2. 在 App 顶部导入入口一次选择这两个文件，CSV 提供 ID/Excerpt，HTML 补充层级和 Highlights。
3. 在“收藏库”中编辑，在“导入并去重”中用历史番号索引排除已有内容。
4. 从收藏库页导出官方 CSV 或 HTML。
5. 需要回到 Raindrop 时，优先导入 CSV；HTML 可用于需要书签层级兼容的场景。

CSV 字段对应关系：

| App 字段 | Raindrop CSV 字段 |
| --- | --- |
| Collection | `folder` |
| Link | `url` |
| Title | `title` |
| Note | `note` |
| Tags | `tags` |
| Created | `created` |
| Cover | `cover` |
| Highlights | `highlights` |
| Favorite | `favorite` |
| Raindrop ID | `id` |
| Excerpt | `excerpt` |

### 24.1 Collection 默认规则

| 状态 | 默认 Collection |
| --- | --- |
| `not_found` | `需要手动核验` |
| 其他状态 | `MissAV_Import` |

你可以在收藏库或整理卡片中手动覆盖 Collection。

### 24.2 Tags 默认规则

如果 Raindrop Tags 字段为空，导出时会根据本地 tag 自动生成：

- 有女优 tag：加入女优 tag。
- 没有女优 tag 且状态异常：加入 `#未知女优`。
- 状态不是 `not_found`：加入类型 tag。
- 状态是 `not_found`：加入 `需要查找`。

如果你不想导出自动 tags，可以在 Raindrop Tags 字段里手动写入最终 tags。

## 25. 常见问题

### 25.1 为什么 `MESSAGE-13` 被识别的问题已经要过滤？

当前版本已经把 `MESSAGE`、`MESSAGES` 等 HTML 噪声前缀列入过滤。一般不会再把 Telegram message id 当番号。

### 25.2 为什么 `moodyz` 不应作为番号？

`moodyz` 是厂牌词，不是番号。当前版本已把 `MOODYZ` 放入噪声前缀过滤，避免从网页标题或评论中误提取。

### 25.3 链接明明能打开，为什么还有提醒？

可能原因：

1. 数据库状态仍是 `not_found`。
2. 页面能打开，但页面内容太短或没有提取到可靠标记，所以状态是 `need_manual_check`。
3. Raindrop Tags 或 Collection 缺失，导出预览仍会提醒。

解决方式：

```text
数据 -> SQLite 本地库 -> 整理卡片 -> 状态核验
```

打开记录，确认链接后把状态改为 `ok` 或 `need_manual_check`，保存。

### 25.4 什么时候会加 `需要查找` tag？

只有记录状态是 `not_found` 时，自动 tags 才会加入 `需要查找`。如果该链接已经确认可访问，请不要让状态停留在 `not_found`。

### 25.5 批量改错了怎么办？

进入：

```text
数据 -> SQLite 本地库 -> 备份恢复
```

选择修改前的备份恢复。恢复前 App 会自动创建一份“恢复前备份”。

### 25.6 CSV 工作台和 SQLite 本地库有什么区别？

SQLite 本地收藏库是长期主数据库，并额外维护不会因删除收藏而消失的历史番号索引；CSV 工作台直接编辑单个外部备份文件。两者使用同一套 Raindrop 官方字段，可以互相导入和导出，但长期整理、搜索和去重建议在本地收藏库完成。

### 25.7 可以直接修改全部数据库内容吗？

可以，进入“原始表”。但更推荐先用“收藏库”“女优 Tag”“类型 Tag”“整理卡片”这些安全入口。原始表适合高级修复。

## 26. 最佳实践

1. 每次大批量导入前，先创建数据库备份。
2. 粘贴 Telegram HTML 后，先看识别数量是否异常。
3. 处理完成后，不要急着导入 Raindrop，先看“导出预览”。
4. 对“有链接但状态异常”的记录，优先用“整理卡片 -> 状态核验”处理。
5. 对 tags 混乱的数据，先修本地女优/类型 tag，再写入 Raindrop Tags。
6. CSV 只作为兼容格式，主整理尽量在 SQLite 本地库完成。
7. 原始表修改前一定备份。

## 27. 快速排错清单

| 问题 | 去哪里处理 |
| --- | --- |
| 识别出了奇怪番号 | 处理页先复制识别结果检查；必要时告诉开发者补噪声词 |
| 无链接太多 | 数据体检 -> 为无链接生成链接；或整理卡片 -> 链接问题 |
| 可访问链接被当作未找到 | 整理卡片 -> 状态核验，改状态并保存 |
| 缺女优 tag | 收藏库详情或整理卡片中补女优 Tag |
| Tags 不想用自动结果 | 在 Raindrop Tags 字段手动写入最终 tags |
| Collection 不对 | 收藏库或整理卡片修改 Collection；也可批量设置 |
| CSV 格式混乱 | CSV 工作台打开、校验、规范番号、另存为 |
| 数据库改坏 | 备份恢复 |

## 28. 建议的日常工作节奏

如果每天都要整理新数据，推荐固定流程：

1. 打开 App。
2. 处理页粘贴 Telegram 导出 HTML 或导入文件。
3. 看识别数量是否合理。
4. 开始处理。
5. 进入“整理卡片”，先处理“链接问题”和“状态核验”。
6. 进入“导出预览”，确认问题数量。
7. 导出 Raindrop CSV。
8. 导入 Raindrop。
9. 创建一次数据库备份。

这样可以最大程度减少误判和重复返工。
