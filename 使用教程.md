# TG 内容工具箱使用教程

> 适用版本：本地 `dist\TG_Content_Toolbox_v0.3.0.exe`
>
> 当前定位：统一接收 Telegram 指定群组，再按群绑定分别运行推特博主、Bad.news、MissAV 和 123AV 四个独立工具。

## 1. 软件现在管理什么

软件长期维护的是“番号记录”，不是本地收藏夹。

每条番号记录主要包含：

- 标准化番号；
- 最佳 MissAV 链接；
- 查询状态；
- 女优 tag；
- 类型 tag；
- 创建时间和处理历史。

旧版本产生的本地收藏、Favorite、Collection 等兼容数据不会被自动删除，但当前界面不再把它们当成本地收藏管理器。Raindrop HTML / CSV 仍可从“本次处理结果”导出；也可以从独立“同步”页面把当前批次直接写入远端账号。

## 2. 启动与数据位置

直接双击：

```text
dist\TG_Content_Toolbox_v0.3.0.exe
```

软件是便携版，但数据库不放在 EXE 旁边，而是保存在 Windows 用户数据目录。升级或替换 EXE 不会覆盖现有数据库。

处理页的“导入番号文件”支持一次多选 TXT、HTML、HTM、MD、JSON、CSV 和 LOG。多个文件内容会合并后统一提取并去重；其中某个文件读取失败时，其余成功文件仍会导入，界面会显示失败数量。

在“日志、备份与外观 -> 数据位置”可查看实际路径。需要放进项目文件夹时，先在项目中建立一个专用空文件夹，再点“更改位置”并选中它；软件会自动备份、复制、校验并重启，原数据库不会删除。进行大批量删除、批量改状态或恢复操作前，建议先到“数据中心 -> 维护 -> 备份恢复”创建备份。

## 3. 推荐日常流程

```text
TG 自建群组中的收藏/转发内容
  -> 首次用每个群的官方导出文件建立历史底库
  -> 在“Telegram 来源”选择最多 5 个群
  -> 到每个工具顶部绑定它对应的群（同一群可绑定多个工具）
  -> 推特 / Bad.news：选择分钟范围，过滤后复制或导出 TXT
  -> MissAV / 123AV：分别过滤番号、创建各自批次、手动启动查询
  -> MissAV 抽查标签，在 Raindrop 页预览 missav1 / missav2 后手动同步
  -> 123AV 在账号操作页手动收藏或仅导出
```

推特与 Bad.news 的输入和结果只在当前软件会话中保留，不建立永久结果库。Telegram 群来源、消息指纹和断点仍保存在数据库中，避免增量更新重复读取。MissAV 与 123AV 各自建立批次：MissAV 批次只启用查询与 Raindrop 任务，123AV 批次只启用查询与收藏任务。

### 3.0 四个工具怎么用

先从侧栏进入“工具首页”，再选择工具：

1. `推特博主`：粘贴内容或一次导入多个文件；可绑定群组并设置开始/结束分钟。输出“博主名”和“主页链接”两个文本框，分别一行一个。
2. `Bad.news 帖子`：只输出 `https://bad.news/t/数字`；`/app`、官网其他页面、查询参数与网址片段都会被移除。
3. `MissAV`：继承原番号过滤、查询、标签、历史库和导出。同步时，如果影片任一女优在本批次创建前已存在于女优库，则目标为根目录 `missav1`；否则为 `missav2`。同批次中新出现的女优不会改变后续记录的判断。
4. `123AV`：继承原查询和账号操作；收藏仍只保留 Chrome 扩展、APP 内执行器、仅导出，远端执行固定单路。

四个工具的时间输入精确到分钟；结束时间包含该分钟的最后一秒。手动粘贴的普通文字没有消息时间，因此不会被时间范围删掉；Telegram HTML/JSON 和增量消息会按自身时间过滤。

### 3.1 Telegram 群组来源的三种方式

打开侧边栏“Telegram 来源”，在页面内选择一种入口：

1. `机器人增量`：推荐用于今后新增消息。只需要在 Telegram 客户端通过 `@BotFather` 创建机器人并复制 Bot Token，不需要 `api_id/api_hash`，也不需要登录 `my.telegram.org`。
2. `个人账号 API`：高级增量入口，可在需要时补读群组历史。先到 `my.telegram.org/apps` 申请自己的 `api_id` 和 `api_hash`。首次在 APP 中填写后，推荐点击“显示登录二维码”，再用手机 Telegram 的“设置 → 设备 → 连接桌面设备”扫码确认；二维码过期会自动刷新。无法扫码时展开“使用手机号验证码登录”，输入手机号、Telegram 验证码和可选的两步验证。成功后会话和 API 凭据由 Windows 安全存储加密，APP 下次启动会自动重连。
3. `官方导出文件`：用于过去历史。在 Telegram Desktop 打开需要处理的群组，使用“导出聊天记录”，格式选择 JSON 或 HTML；每个群分别导出。APP 支持一次选择多个 `result.json`、`messages.html` 或完整导出文件夹。

三种入口共享消息级去重和断点，不会自动启动任何网站查询。接收后，消息会按各工具保存的群组绑定分别投递；同一群可同时投递给 MissAV 和 123AV，或其他工具。来源页仍保留“本次新增番号”预览，便于兼容旧流程。

机器人首次设置：

1. 在 Telegram 客户端打开官方 `@BotFather`，发送 `/newbot` 并按提示创建机器人。
2. 发送 `/setprivacy`，选择该机器人并设为 `Disable`；如果机器人此前已经加入群组，修改隐私设置后将它移除并重新加入。也可以把机器人设为群管理员，使其接收全部消息。
3. 把机器人加入需要处理的至多 5 个自建群。
4. 每个群发送一条不含番号的普通识别消息。
5. 在 APP 的“机器人增量”中粘贴 Token，点击“连接并安全保存”，再点击“发现群组”。
6. 勾选群组并“保存来源”。识别消息作为首次基线，不进入番号结果；以后点“同步全部机器人来源”读取新消息。

Bot Token 经 Windows 安全存储加密，不写入 SQLite 或日志。机器人无法读取加入前的历史消息；Telegram 对尚未被任何接收端取走的机器人更新只临时保留最长约 24 小时，因此建议每天至少同步一次。消息一旦被 APP 写入本地就会永久保存；电脑关闭超过 24 小时时使用官方导出文件补齐。

机器人使用一个全局更新游标，因此所有已选群组一起同步，不提供容易造成漏消息的“只同步此群”。每批最多读取 100 条更新，并且必须先完成消息去重和 SQLite 写入，随后才推进全局游标；停止会等当前批安全写完。

API 连接后点击“加载/刷新群组”，从群组、超级群组和频道中勾选最多 5 个来源，再点击“保存来源”。首次保存时会把每个群当前最新消息记为基线，因此第一次正常同步不会重新扫描旧历史。以后可以同步全部来源，也可以在单个来源旁点“同步此群”。

每个群独立保存基线、最高检查点和跨页继续游标。API 同步可限制每个群本次最多读取 500～20000 条；若新增消息超过上限，下次会从该群断点继续，不影响其他群。中途停止时已经读取的部分仍会安全写入。

“已处理消息回看”只回看首次绑定基线之后最近的一小段内容，用于识别后来编辑过的消息。只有填写“补查起始日期”时才会临时越过基线回补历史空档；回补内容仍经过永久查重。移除来源群只停止后续同步，不删除已有番号、消息指纹或原检查点。

本地 SQLite 只保存来源、消息 ID、日期、内容摘要和提取番号，不保存完整 TG 消息正文；手机号、验证码和两步验证密码也不进入数据库或日志。

扫码登录仍可能在账号启用两步验证时要求输入 2FA 密码；密码只在当前授权内存中使用，不会保存。二维码本身是短时登录凭据，不要截图发给其他人。取消扫码会立即中止后台授权，不保存半成品会话。

如果 Telegram 在手机号登录阶段要求“官方客户端级 reCAPTCHA”，APP 会明确暂停，不会尝试绕过或代填。此时取消登录，优先改用二维码；仍不能登录时使用“官方导出文件”，或稍后更换网络再试。

### 3.2 第一次配置 Raindrop 直接同步

1. 在 Raindrop 的 App 管理页面复制个人 Test Token（自用）或准备 OAuth Access Token。
2. 从工具首页进入 MissAV，再切到工作区“Raindrop”阶段，粘贴令牌并点“验证并保存”。令牌由系统安全存储加密，界面、数据库和日志不保留明文。
3. 选择本次 MissAV 处理批次。新版批次不再手动选择单一 Collection，而是自动准备根目录 `missav1` 与 `missav2`。
4. 点击“生成预览”。软件会先检查远端 URL，并逐条显示目标目录、新建、更新、无需变化和异常。
5. 检查无误后点“开始同步”，再确认一次写入数量。

预览中的“官网已有同一链接（书签 ID #…）”表示 Raindrop 账号中已经有该网址。本次不会重复新建，而会更新它的标题、Tags，并将它放入预览显示的 `missav1` 或 `missav2`。这两个根 Collection 不存在时软件会自动创建。同步明细可拖动下边缘调整高度，或点“展开表格”进入大视图，按 `Esc` 还原。

远端核对最多 100 个 URL 一组；实际写入按官方 API 速率限制平滑执行。程序中断或写入请求返回不明确时，该条会进入“需核对远端”，下次预览会重新检查 URL，避免盲目重复创建。

## 4. 从 TG 内容提取番号

进入“番号处理”。可以：

- 直接粘贴 Telegram 消息、普通文本、Markdown 或网页片段；
- 从“来源”页送入机器人、个人账号 API 或官方导出的新增番号；
- 也可在输入页直接选择 Telegram 导出的 HTML；
- 选择 TXT、MD、CSV 等文件；
- 粘贴 MissAV 完整链接。

软件会：

1. 从文本和链接中识别番号；
2. 统一大小写、连字符等格式；
3. 过滤常见 Telegram 消息编号和噪声；
4. 合并本批次重复项；
5. 与永久历史库比较。

当输入是 Raindrop 官方 CSV 时，软件会启用结构化过滤：只从 `title` 和可信 AV 站点的 `url` 提取番号，不再扫描 `id`、`created`、`cover`、普通网页 URL 和网页 `excerpt`。这样可避免把封面文件名、论坛帖子号、课程年份及 `Office 365` 等产品版本误当成番号。写真编号和同人游戏商品号默认不进入 MissAV/123AV 查询队列。

原始文字下方的“过滤后的番号”框会实时显示最终识别结果，每行一个。工具栏中的复制和保存清单按钮均使用这份过滤结果。

提取后先检查“导入去重”结果，再决定处理哪些记录。

## 5. 导入去重

“导入去重”以永久番号历史为准，不依赖旧收藏数据。

常用处理方式：

- 只保留全新番号：适合日常批次；
- 保留待复查：适合以前查不到、现在可能已经上线的番号；
- 逐条选择：适合混合批次或重点复查。

同一番号已经存在时，软件会优先保留历史信息，避免因为重复导入丢掉状态和标签。

### 5.1 处理批次与断点续跑

开始前可以填写“批次名称”，例如 `TG 7月21日晚`；留空时软件会按时间自动命名。点击开始后，软件会先把过滤后的番号、来源说明、速度和输出目录写入本地数据库，再访问网络。

每条完成后会立即保存查询状态、链接、女优/类型 Tag、错误原因和完成时间。原始 TG 全文不会进入批次表。

手动停止或程序异常退出时：

- 已完成项原样保留；
- 正在处理和尚未开始的项回到“等待处理”；
- 工具首页进入 MissAV 或 123AV 后，其“执行”阶段会显示最近的未完成批次；
- 也可以进入“番号库 -> 处理批次”，查看逐条明细并点击“继续剩余”；
- 继续时沿用该批次原来的速度和输出目录，无法使用原目录时会要求重新选择。

批次页还支持重命名和把任意历史批次重新载入结果表。批次记录不会创建本地 Favorite 或 Collection，也不会重复复制永久 `codes` 主数据。

如果几千条批次只处理了一部分便不再需要，可在“站点”工作台、处理结果或“番号库 -> 处理批次”点击“删除整个批次”。正在运行时软件会先安全停止并等待在途请求结束，再删除批次、逐条结果和四阶段任务。删除前自动建立数据库备份；已经写入永久番号库的记录和已导出文件会保留，避免误删其他批次共用的数据。

从 `v0.1.4` 开始，新批次的每条记录都有四个独立任务；`v0.1.5` 开始实际执行 123AV 查询，`v0.1.6` 起在界面中按两条支线分板块显示，`v0.1.7` 增加可恢复的 123AV 账号收藏，`v0.1.8` 修复已识别账号仍误报未登录，`v0.1.9` 曾改为两站公平并行，`v0.1.10` 再按实际工作方式改成两个站点完全手动分开。MissAV 查到可用页面后，Raindrop 环节变为“待导出”；无论 MissAV 是否命中，123AV 都保留独立待办。`v0.1.3` 及更早批次仍显示为“旧版 MissAV 单支线批次”，升级不会给它们补建待办。

## 6. 开始处理

确认输入后，点击输入页底部的“打开执行工作台”，或使用当前工具顶部的“执行”阶段。在对应网站容器中手动点击“开始 MissAV”或“开始 123AV”。软件只运行你点击的站点，不会自动启动另一个工具。当前站点完成后，可以立即检查结果，也可以回工具首页启动另一站。已经存在于永久历史中的番号可以跳过 MissAV，但仍可单独建立 123AV 查询任务。

输入与运行已经分成两个页面：“处理”只负责原文、文件导入和番号过滤；“站点”负责批次、网站任务、速度、异常队列、输出目录、进度和 ETA。站点页左侧是二级目录，右侧一次只展示 MissAV、123AV 或批次管理中的一个详情页；以后接入新网站时，只在目录增加入口，不会把所有网站配置堆在同一屏。

123AV 只接受 URL、标题或详情字段中出现精确番号的详情页面，不再通过搜索页找模糊结果。正常详情页中的相关影片卡片不会影响详情识别；所有已选详情变体均未命中时才判为“未找到”。登录页、无法识别的页面进入人工核验，403/429、限流和验证页进入网络错误。

处理开始后，“站点”工作台和“处理结果”页都会显示预计剩余时间板块。完成首条记录后，软件根据最近最多 20 条的实际速度估算预计剩余、预计完成时间、已用时间和单条平均耗时；网络速度变化时会自动调整。刚开始尚无样本时显示“计算中”，停止或完成后会显示最终状态。

站点工作台中的 MissAV 和 123AV 各有独立状态卡、开始/继续按钮、停止按钮、速度配置和专属设置。进度与预计时间只计算当前手动启动的站点；完成后卡片会显示本次条数、实际用时和每秒处理量。

123AV 站点内部再分成两个子页面：

- “番号查询”只负责公开详情页查询、查询速度和网络错误；
- “账号收藏”只负责在本地 Chrome 的已登录环境中收藏查询命中的影片，不会自动启动番号查询；
- 查询与收藏完全独立。停止收藏不会停止 123AV 查询；登录失效、CAPTCHA、Error 1015 或状态不明只进入收藏结果；
- 软件重启时只自动恢复只读查询。曾处于执行中的收藏转为“需核对远端”，避免不确定状态下重复点击。

开始前可分别在“MissAV 速度”和“123AV 速度”选择七种模式。两个选择独立保存、互不覆盖；续跑旧批次时会恢复该批次分别记录的两站配置：

| 模式 | MissAV 行为 | 123AV 工作路数 |
| --- | --- | --- |
| 稳妥 | 1 路，完整检查候选 | 1 路 |
| 智能（MissAV 默认） | 2 路起步、最高 3 路，完整检查候选 | 最高 3 路 |
| 高速 | 4 路，完整检查候选 | 4 路 |
| 极速 6 | 6 路，只查 MissAV 主地址 | 6 路 |
| 飞速 8（123AV 默认） | 8 路，只查 MissAV 主地址 | 8 路 |
| 狂飙 12 | 12 路，只查 MissAV 主地址 | 12 路 |
| 极限 16 | 16 路，只查 MissAV 主地址 | 16 路 |

速度模式在处理开始后会锁定，完成或停止后才能切换；两个站点的选择都会保留到下次启动。MissAV 的档位控制候选数和并发，123AV 的档位控制详情候选和工作路数；两站的真实发送速度都在各自下方“请求调度目标”中独立控制。稳妥、智能和高速档会检查完整候选；极速 6、飞速 8、狂飙 12 和极限 16 使用精简候选。123AV 所有档位都不使用搜索页。

MissAV 与 123AV 的实际请求速率都提供两种方式：

- “自动探速”（默认、推荐）：MissAV 首次约从 8 RPS、123AV 约从 5 RPS 起步，以后分别读取本站上次学到的稳定值；真实延迟和完成速度健康时逐级升速，默认最高 16 RPS。
- “固定速率”：按所选目标平滑发送，适合已经确认某个速率长期稳定时使用；硬总超时和 HTTP 429 紧急冷却仍然生效。

两站上限滑杆范围都是 1–32 RPS，并提供 4、8、12、16、24、32 六个快捷值。速率上限和自动/固定方式可在本站运行中调整，不必停止批次；16 路工作并发可以配合任意 RPS 上限。界面区分“调度目标 RPS”“实际网络请求完成速度”和“实际番号/秒”，缓存命中也会单独计数，避免把目标值误当成真实吞吐。两站的模式、上限和学习值完全分开保存。

自动模式不再只观察有没有 429：最近响应的 p95 延迟过高、硬超时、连接重置、Cloudflare/验证码页面或错误比例升高，都会触发本站平滑降速。正文读取超过速度档位总时限会被真正中止；Electron 请求失败后不会再暗中改用 Node 重发一次，因此少量慢请求不会长时间占满全部工作路。

MissAV 的网络错误策略与 123AV 使用同类控制：

- “稳定自适应”（默认）：延迟或错误升高时降低新请求压力；
- “只跑一轮”：尽量保持工作路，错误留到结果页集中重跑；
- “动态半速”：高档位发生异常时最低保留所选工作路的一半。

123AV 的 6–16 路档位另有“网络错误策略”：

- “分层收尾”（默认、推荐）：主轮完成后自动用较少工作路重跑网络错误；成功项不会重复请求。
- “保持并发，错误后单独重跑”：所选并发保持不变，单条超时、429 或访问异常留在 123AV 网络错误队列；整批跑完后使用结果页的重跑按钮集中复查。适合短批次或确定站点没有限流时追求最快首轮吞吐。
- “动态半速”：普通网络异常时将工作路最低保留在所选值的一半。

三种策略都共享同一个 429 断路器：首个 429 返回后，还没发出的请求立即暂停，优先遵循 `Retry-After`，否则默认冷却约 10.5 秒；当前 RPS 回退到约 80%，同一批次不再超过本次失败速率的约 90%。自动模式至少保留 3 RPS，只有经过更长的稳定区间才会重新探高，不会再把 16 路粗暴变成 1 路。

因为 MissAV 需要解析详情页和标签、还可能检查多个候选地址，它和 123AV 的实际每秒速度不能直接按同一并发档位等同比较。两个面板功能相同，但学习值、错误策略和实际吞吐互不影响。

速度档位表示当前手动启动站点的并发上限。例如“极限 16”表示当前 MissAV 或当前 123AV 最多 16 路，另一站不会自动占用并发。这样单站通常更快、更容易判断异常；如果最终把两个站都顺序跑完，总耗时可能比旧版并行执行略长。

常见状态：

| 状态 | 含义 |
| --- | --- |
| `ok` | 找到可用页面 |
| `not_found` | 当前未找到 |
| `network_error` | 网络、限流、防爬或临时访问异常 |

成功找到页面时，软件会抓取女优 tag 和类型 tag，并写入永久番号库。

网络超时、403/429、访问验证和防爬页会保存为 `network_error`，不会混入“页面待核验”，也不会进入本批 Raindrop 导出。之后再次导入同一番号时会自动允许重试。

正常详情页自身可能加载验证码或访问控制脚本。软件会同时核对目标番号和详情结构，不会再仅因源码中出现 `captcha` / `challenge-platform` 字样而误报网络错误。成功确认详情页后也会停止请求其他候选地址，以减少大批量处理时间。

“页面待核验”只表示 MissAV 页面成功返回、正文明确包含目标番号，但内容太短或缺少播放器/女优/类型等详情证据。候选地址只是能打开，不再足以判定为待核验。

处理任一站点时手动停止，已完成的查询原样保存，尚未开始或中断的当前站点只读查询回到队列；另一站不受影响。再次继续时必须明确选择“继续 MissAV”或“继续 123AV”。

## 7. 本次处理结果

结果页用于检查当前批次，不等同于整个本地库。

可进行：

- 查看每条番号状态、链接和标签；
- 使用“MissAV / 123AV”板块切换查看结果；MissAV 板块只显示查询、Raindrop 与标签，123AV 板块只显示查询与收藏状态；
- 搜索番号、链接、状态、标签或备注，并按状态、最终标签和页签组合筛选；
- 点击番号、状态或最终 Tags 表头排序，也可使用排序下拉框；
- 单击整行选择，使用 Ctrl 切换选择、Shift 连选、Ctrl+Shift 追加区间；
- 把所选记录或当前筛选结果复制成 TSV，直接粘贴到 Excel / WPS；
- 对 `network_error` 使用行内“重新运行”，或批量重跑选中/全部网络错误；批量按钮只重跑当前站点板块的查询；
- 在“站点 -> 123AV -> 账号收藏”选择批次并发起收藏；结果页只保留与当前记录相关的收藏状态和快捷入口；
- 对“需核对远端”使用单条、所选或全部复查；复查只读远端状态，不会直接再次点击收藏；
- 导出全部结果、当前所选/筛选范围、仅 HTML、仅 CSV，或按最终标签拆分导出。

网络错误重跑使用当前速度档位。123AV 重跑直接写回原批次，只改变 123AV 查询及其收藏依赖，不覆盖 MissAV 链接、标签或 Raindrop 状态。停止时尚未完成的查询回到队列，之后可继续。

Raindrop 导出工具只出现在 MissAV 板块。123AV 板块复制出的 TSV 也只包含 123AV 查询、链接、收藏状态和本站错误，避免把两个站点的数据列再次混到一起。

### 7.1 123AV 账号收藏

入口位于“站点 -> 123AV -> 账号收藏”。它与“番号查询”完全分开，进入页面或完成查询都不会自动收藏，除非你提前开启“查询结束后自动收藏”。同一时间 123AV 只能执行查询或收藏之一；MissAV 等其他网站使用自己的队列，仍可同时工作。

收藏方式有三种：

1. `Chrome 扩展`：复用本地 Chrome 已登录账号，通过本机桥接执行。
2. `APP 内执行器`：使用 APP 独立账号窗口；固定单路，网络异常、Error 1015 或状态不明时休息 10 秒后自动继续。
3. `仅导出`：不访问账号，生成一行一个番号的 TXT 和带详情链接/状态的 CSV。

首次使用本地 Chrome 扩展桥：

1. 点击“安装/配对扩展”，APP 会打开扩展文件夹并把配对码复制到剪贴板。
2. 在 Chrome 打开 `chrome://extensions`，开启“开发者模式”，点击“加载已解压的扩展程序”，选择 APP 打开的文件夹。
3. 点击扩展图标，粘贴配对码并连接。以后正常启动 APP 和 Chrome 后会自动重连，无需每批重复配对。
4. 确保当前 Chrome 已登录 123AV，再回到 APP 点击“检查登录”。

Chrome 扩展和 APP 内执行器都固定 1 路串行。每条记录先核对详情页可见番号，再检查“保存/已保存”：已经显示“已保存”会直接跳过；只有明确显示“保存”才点击，点击后必须再次看到“已保存”才算成功。状态不明、导航错误或 Error 1015 会把该项放到主轮结束后的第 2 轮，最多再打开一次。

扩展和 APP 之间只通过 `127.0.0.1` 本机桥接，使用随机配对密钥认证。扩展只读取页面中可见的账号数字、番号、标题和保存状态，不读取密码、Cookie、Local Storage、Session Storage 或完整页面 HTML。登录失效或 CAPTCHA 会暂停收藏。程序中断时，正在执行的收藏转为“需核对远端”，不能在重启后盲目再次点击。

“按标签拆分”优先使用当前选中项；没有选中项时使用当前搜索、页签、状态和标签筛选范围。每个最终 Tag 生成一份 HTML 和 CSV，同时生成 `标签导出索引.csv` 与当前范围处理报告。记录包含多个 Tag 时会进入多个对应文件，这是为了便于按标签分别导入或归档。

Raindrop 导出只包含本次结果，适合导入指定 Raindrop Collection。软件不再维护本地 Favorite 或 Collection 树。

结果表以及番号库、标签、维护和 CSV 等内部表格都支持拖动表头右边界调整列宽、双击边界自动适配、单击单元格定位、方向键移动和 Ctrl+C 复制。各业务表原有的搜索、筛选、整行多选和编辑能力继续保留。

整条记录都是可点击区域，不必精确点击复选框。

单击记录中的按钮、输入框或下拉框时不会误触发行选择。

- 番号；
- MissAV 链接；

### 8.4 批量操作

选中多条后可：

- 批量改状态；
- 批量添加或移除女优 tag；

- 生成缺失的 MissAV 链接；
- 规范化番号；
- 删除永久番号记录。

批量删除会删除番号及其标签关联，但不会主动清理旧版本保留的兼容表。删除前仍建议先创建数据库备份。

## 10. 维护

维护区集中放置低频工具：

- 数据概览：查看番号与标签规模；
- 数据体检：检查缺链接、缺标签、状态矛盾和坏关联；
- 备份恢复：创建或恢复 SQLite 备份；
- 重复项：检查疑似重复番号；
- 底层数据：查看当前公开的本地库表；
- 处理批次：独立页签查看来源、进度、逐条结果并继续未完成任务；
- 运行日志：查看数据库、导入、MissAV/123AV 网络查询、状态判断、导出和程序错误，支持复制诊断报告、导出日志和打开日志目录。
- 正式启用前归零：先创建完整备份，再清空当前和历史版本的全部业务数据，并重置自增序号、压缩和校验数据库。

需要反馈问题时，优先进入“番号库 -> 维护 -> 运行日志”，点击“复制诊断报告”后直接粘贴给 Codex。日志不保存原始 TG 全文，只记录文件名、番号、URL、状态码、判定结果和错误信息。

高级数据默认打开 `codes`。当前业务表按核心数据、关系数据、任务历史、缓存与同步、Telegram 分类；站点查询缓存和 Raindrop 远端映射也可查看和维护。表格支持单击、Ctrl、Shift、Ctrl+Shift 选择，逐格编辑、批量修改、批量删除、复制 TSV 和导出 CSV。番号唯一性、关联主键和 JSON 字段会在写入前校验。旧版 `bookmarks` 和 `bookmark_collections` 不在日常界面公开，以免继续混用收藏模型。

## 11. 备份与恢复

建议在以下操作前创建备份：

- 大批量删除；
- 大批量改状态或标签；
- 导入大批历史数据；
- 恢复其他数据库前。

恢复会替换当前数据库状态。务必确认备份时间、文件名和记录数量，不要把临时测试数据库恢复到正式数据目录。

### 11.1 正式启用前清空全部数据

进入“番号库 -> 维护 -> 备份恢复”，点击“备份并清空全部业务数据”，再按提示输入 `清空全部数据`。软件会依次：

1. 创建“正式启用前完整备份”；
2. 清空永久番号、女优/类型 Tag 和关系；
3. 清空处理批次、逐条结果和四阶段任务；
4. 清空站点查询缓存、Raindrop 远端同步映射；
5. 清空 Telegram 来源、消息指纹和导入历史；
6. 清空旧版本遗留的收藏或账号运行表；
7. 重置自增 ID、压缩 SQLite，并执行完整性检查。

不会删除数据库结构、备份文件、外观设置，也不会删除 Windows 安全存储中的 Chrome 配对密钥、Raindrop Token、Telegram Bot Token、API 凭据和加密会话。任何查询、收藏或同步任务运行时，归零按钮会拒绝执行。

## 13. 常见问题

### 为什么升级 EXE 后数据还在？

数据库保存在 Windows 用户数据目录，和便携 EXE 分开。这是正常设计。

### 为什么停止后还有“未完成批次”？

停止现在表示暂停，不是丢弃。已完成结果和剩余队列已经写入数据库，可以在“站点”工作台或“番号库 -> 处理批次”继续；只有全部条目结束后批次才会标记为已完成。

### 为什么数据库里可能还有 bookmarks 表？

这是旧版本兼容数据。当前版本不再自动创建收藏记录，也不在界面中提供收藏管理。为避免破坏用户历史数据，升级不会自动删表或清空。

### 为什么导出只看到本批次？

当前 Raindrop 导出是处理结果的下游输出，不是整个旧收藏数据库的导出。

### 单击记录为什么没有打开新页面？

单击现在用于选择记录并显示右侧详情，行为与文件管理器选择文件一致。

### Shift 连选从哪里开始？

从最近一次普通单击或 Ctrl 单击建立的锚点开始。没有锚点时，当前记录会作为起点。

### 可以直接删除旧收藏表吗？

不建议。旧表可能仍有需要恢复的信息。当前版本只是停止使用和展示，不会擅自销毁。

## 14. 数据安全原则

- 以本机数据库为准，不以 Git 或远端仓库判断用户数据状态；
- 以 `dist` 中实际生成的最新 EXE 作为本地发布版本；
- 正式数据库只通过应用界面管理；
- 开发和测试使用临时数据库；
- 不确定时先备份，再修改。
