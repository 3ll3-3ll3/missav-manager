# TG 内容工具箱 v0.3.0

TG 内容工具箱是一个本地桌面 App。它统一接收最多 5 个 Telegram 群组，再按工具绑定把消息分别送入“推特博主”“Bad.news 帖子”“MissAV”“123AV”四个独立入口。推特与 Bad.news 只生成当前会话的一行一个文本结果；MissAV/123AV 继承原有查询、标签、收藏、数据库、Raindrop 和导出能力。

本 README 已包含从首次启动到数据库维护的完整操作教程；需要更细的状态说明和背景解释时，可继续阅读：[使用教程.md](使用教程.md)。

开发与 agent 接手请先阅读：[项目交接说明](docs/PROJECT_HANDOFF.md) 和 [2026-07-12 修改日志](docs/CHANGELOG_2026-07-12.md)。

## 给 Windows 用户

当前本地最新版以 `dist` 目录中实际生成的最新便携版 EXE 为准：

```text
dist\TG_Content_Toolbox_v0.3.0.exe
```

1. 找到上述单个 EXE；需要时可复制到任意可写文件夹。
2. 双击 EXE 启动，无需安装 Node.js、Git 或其他运行环境。
3. 首次运行会自动创建自己的本地数据库；升级 EXE 不会覆盖已有数据。

便携 EXE 约 400 MB，首次启动需要先解压并接受系统安全扫描，在部分机器上可能等待约一分钟；看到窗口后后续启动通常会更快。

这是未签名的个人桌面软件，Windows SmartScreen 可能会显示提示。请使用本项目 `dist` 目录中确认过的构建文件。

应用默认把数据库和备份保存到 Windows 用户数据目录。你也可以在“日志、备份与外观 -> 数据位置”选择项目文件夹里的专用空目录；软件会先备份、复制并校验数据库，保留原文件，然后重启到新位置。

处理页面时需要网络能够访问 MissAV 和 123AV。123AV 公开查询使用 APP 自己的独立网络会话；账号收藏可选择本地 Chrome 扩展、APP 内执行器或仅导出清单。Chrome 扩展只读取页面中可见的账号、番号、标题和保存状态，不读取密码、Cookie、Local/Session Storage 或完整页面 HTML。

## 给开发者

需要 Node.js 22.12 或更高版本：

```bash
npm install
npm start
```

打包便携版：

```bash
npm run build:portable
```

该命令默认使用国内镜像、禁用签名并在打包前检查应用是否仍在运行。为避免 NSIS 高压缩在部分 Windows 环境长时间卡住，当前采用快速封装；生成的单文件 EXE 会比高压缩版本更大。

开发模式：

```bash
npm run dev
```

```text
群组历史导出或 Bot/API 增量 -> 消息按群绑定进入对应工具 -> 推特/Bad.news 直接复制或导出 -> MissAV/123AV 分别建立自己的批次并手动运行 -> 检查异常 -> 123AV 手动收藏 -> MissAV 在 Raindrop 页预览 missav1/missav2 分流后手动同步
```

## 完整使用教程

### 1. 第一次启动

1. 双击 `dist\TG_Content_Toolbox_v0.3.0.exe`。首次启动可能因解压和 Windows 安全扫描等待约一分钟。
2. 进入“日志、备份与外观 -> 数据位置”，确认当前数据库和备份目录。
3. 如果准备长期使用，先进入“番号库 -> 维护 -> 备份恢复”创建一次空库备份。
4. 如果旧版本已有测试数据，需要从零开始，可在没有任何任务运行时选择“备份并清空全部业务数据”，输入 `清空全部数据` 确认。软件会先备份再清空，账号令牌和备份文件不会删除。
5. 从侧栏进入“工具首页”。四个工具彼此独立，先打开需要的工具，不必同时配置或运行全部工具。

软件默认使用护眼淡绿色主题且不加载背景图。更换 EXE 不会自动删除数据库；数据位置与 EXE 位置彼此独立。

### 2. 选择输入方式

每个工具都支持以下输入来源：

- 在工具的输入框直接粘贴文字；
- 一次选择多个 TXT、HTML、HTM、MD、JSON、CSV 或 LOG 文件；
- 导入 Telegram Desktop 导出的 HTML、JSON 或完整导出目录；
- 从已经配置的 Telegram Bot 或个人账号 API 接收指定群组的增量消息。

包含时间的 Telegram 消息可设置开始和结束分钟，结束分钟包含该分钟的最后一秒。普通粘贴文字没有消息时间，因此不会被时间范围误删。多个文件会合并过滤、规范化和去重；某个文件读取失败不会阻止其他文件导入。

### 3. 配置 Telegram 增量来源

日常增量优先推荐机器人；旧历史优先使用 Telegram Desktop 导出文件；个人账号 API 作为需要回读更多历史时的高级入口。

#### 3.1 Bot 方式（推荐）

1. 在 Telegram 中打开官方 `@BotFather`，发送 `/newbot` 创建机器人并复制 Bot Token。
2. 向 `@BotFather` 发送 `/setprivacy`，选择该机器人并设为 `Disable`。若机器人已经在群中，修改后将它移出再重新加入；也可直接把机器人设为群管理员。
3. 把机器人加入需要处理的自建群组，最多选择 5 个群。
4. 每个群发送一条不含目标内容的识别消息。
5. 在 APP 的“Telegram 来源 -> 机器人增量”粘贴 Token，点击“连接并安全保存”，然后点击“发现群组”。
6. 勾选群组并保存来源。首次识别消息会成为基线，不会进入过滤结果。
7. 到各工具的“来源绑定”中选择它负责的群。同一个群可以同时绑定多个工具。
8. 日后点击“同步全部机器人来源”，新消息会按绑定关系分别送入工具。

机器人看不到加入群组之前的历史；尚未由接收端取走的更新也只会由 Telegram 临时保留，因此建议每天至少同步一次。错过的历史用 Telegram Desktop 导出文件补齐。Bot Token 由 Windows 安全存储加密，不写入 SQLite 或运行日志。

#### 3.2 Telegram Desktop 历史导出

1. 在 Telegram Desktop 打开目标群，选择“导出聊天记录”。
2. 选择 HTML 或 JSON 格式，每个群分别导出。
3. 在 APP 的“Telegram 来源 -> 官方导出文件”一次选择多个 `messages.html`、`result.json`，或直接选择完整导出目录。
4. 预览群名、消息时间和新增数量，确认后导入。
5. 再到对应工具设置群绑定和时间范围。

导出文件、Bot 和个人账号 API 共用消息指纹去重。APP 只长期保存来源、消息 ID、日期、摘要和提取结果，不保存完整 Telegram 正文。

#### 3.3 个人账号 API（可选）

1. 在 `my.telegram.org/apps` 申请自己的 `api_id` 与 `api_hash`。
2. 在“Telegram 来源 -> 个人账号 API”填写凭据，优先选择二维码登录。
3. 用手机 Telegram 的“设置 -> 设备 -> 连接桌面设备”扫码；若账号启用了两步验证，再输入 2FA 密码。
4. 登录成功后加载群组，选择最多 5 个来源并保存。
5. 首次保存会以群组当前最新消息为基线，之后按群独立增量同步。

API 凭据和会话由 Windows 安全存储保护。二维码、验证码、手机号和 2FA 密码不会写入日志或数据库。若登录被 Telegram 风控，先使用 Bot 和官方导出文件，不要连续重复申请。

### 4. 推特博主工具

1. 从“工具首页 -> 文本工具 -> 推特博主”进入。
2. 粘贴消息、导入文件，或选择已经绑定的 Telegram 群组。
3. 需要时设置开始/结束时间，点击过滤。
4. 检查“博主名”和“主页链接”两个结果框。
5. 分别复制或导出 TXT；每行一项，链接统一为 `https://x.com/用户名`。

该工具识别 `@handle`、`#handle`、x.com 和 twitter.com 主页。结果仅保留在当前会话，不建立永久博主数据库。

### 5. Bad.news 帖子工具

1. 从“工具首页 -> 文本工具 -> Bad.news 帖子”进入。
2. 粘贴消息、批量导入文件或选择绑定群组。
3. 设置可选时间范围并点击过滤。
4. 复制或导出一行一个的链接。

只保留规范形式 `https://bad.news/t/数字`。查询参数、锚点、`/app`、下载页和官网其他页面会被去除。结果同样只保留在当前会话。

### 6. MissAV 工具

#### 6.1 过滤并建立批次

1. 从“工具首页 -> 影片工具 -> MissAV”进入。
2. 粘贴 Telegram 内容、MissAV 链接或番号；也可一次导入多个文件。
3. 检查“过滤后的番号”文本框，内容应为一行一个标准番号。
4. 在“导入去重”选择只保留全新项、包含待复查项，或逐条选择。
5. 填写批次名称；留空时软件按时间自动命名。
6. 点击创建 MissAV 批次。创建批次只建立任务，不会自动启动 MissAV 或 123AV。

#### 6.2 查询、速度与异常

1. 打开当前工具的“执行”阶段，手动点击“开始/继续 MissAV”。
2. 初次使用建议采用“智能 + 自动探速”。已验证网络稳定后，可切换更多工作路或提高 RPS 上限。
3. 运行中观察真实番号/秒、网络请求速度、ETA 和错误数量。调度目标不等于实际完成速度。
4. 可随时安全停止；已完成项保留，剩余项回到待处理，下次手动继续。
5. 完成后进入“结果”，抽查页面链接、女优 Tag、类型 Tag 和状态。
6. `network_error` 可单条、选中范围或全部重跑；`not_found` 表示所有已知候选均未命中。

MissAV 命中后会抓取并清洗标签，写入永久番号库。高速档可能只查询主地址；需要尽量减少漏检时使用完整候选的稳妥、智能或高速档。

#### 6.3 Raindrop 手动同步

1. 在 Raindrop App 管理页面准备个人 Test Token 或 OAuth Access Token。
2. 打开 MissAV 的“Raindrop”阶段，粘贴令牌并点击“验证并保存”。
3. 选择已经完成或部分完成的 MissAV 批次，点击“生成预览”。
4. 检查每条记录的目标目录、新建/更新状态和 Tags。
5. 确认后点击“开始同步”；软件不会在查询完成后擅自同步。

批次创建时会冻结当时的已知女优底库。影片任一女优已经存在于该快照时进入根 Collection `missav1`，否则进入 `missav2`；目录不存在会自动创建。远端已有相同 URL 时更新原书签，不会重复新建。

### 7. 123AV 工具

#### 7.1 番号查询

1. 从“工具首页 -> 影片工具 -> 123AV”进入。
2. 输入过滤后的番号并创建 123AV 批次。
3. 打开“执行 -> 番号查询”，手动点击“开始/继续 123AV”。
4. 初次建议“飞速 8 + 自动探速”；网络稳定后再调整工作路和 RPS。
5. 查询只访问详情地址及已知详情后缀，不使用站内搜索页；页面必须出现精确番号才算命中。
6. 在结果页重跑 `network_error`，不会覆盖 MissAV 的链接或标签。

MissAV 与 123AV 是两个独立工具，不会因创建一个批次而自动运行另一个。不同网站可以并行工作；同一网站的查询与账号收藏不会并发争抢。

#### 7.2 账号收藏

查询命中后进入“执行 -> 账号收藏”，选择以下方式之一：

- `Chrome 扩展`：复用本地 Chrome 的 123AV 登录状态；
- `APP 内执行器`：使用 APP 独立账号窗口；
- `仅导出`：不访问账号，只生成番号 TXT 和详情 CSV。

Chrome 扩展首次配对：

1. 在 APP 点击“安装/配对扩展”，软件会打开扩展目录并复制配对码。
2. Chrome 打开 `chrome://extensions`，启用“开发者模式”。
3. 点击“加载已解压的扩展程序”，选择 APP 打开的目录。
4. 打开扩展，粘贴配对码并连接。
5. 确保 Chrome 已登录 123AV，再回 APP 检查账号并开始收藏。

Chrome 与 APP 内执行器均固定单路串行。同一条先核对可见番号和保存状态：已经“已保存”就跳过，只有明确显示“保存”才点击，点击后再次确认“已保存”才算成功。Error 1015、网络异常或状态不明会等待 10 秒并进入后续轮次；登录失效或验证码会暂停，需用户手动处理。

### 8. 结果表与批次管理

- `Ctrl + 单击`：切换某一行；

### 9. 番号库与高级数据编辑

1. 打开“番号库 -> 番号管理”，按番号、链接、状态或标签搜索。
2. 单击记录后在详情区修改番号、MissAV 链接、状态、女优 Tag 或类型 Tag。

4. “标签管理”可查看标签引用数量、番号示例并进行维护。
5. “维护 -> 高级数据”按核心、关系、任务、缓存/同步和 Telegram 分类显示底层表，支持逐格编辑、批量修改/删除、复制 TSV 和导出 CSV。
6. 大批量修改或删除前先创建备份。唯一键、关联主键和 JSON 字段会在写入前校验。

番号库基于原生 SQLite、WAL 和数据库端分页，已通过 100,200 条隔离数据压测。推特与 Bad.news 不写入永久业务库；MissAV、123AV 批次、标签、同步映射和 Telegram 指纹会持久保存。

### 10. 修改数据库位置

2. 在项目目录或其他磁盘创建一个专门的空文件夹；不要选择项目根目录或 `dist`。

迁移不会删除原数据库。确认新位置工作正常并另有备份后，再自行决定是否清理旧副本。不要把正在使用的数据库、备份、日志或令牌提交到 Git。

## 常见问题与故障处理

### APP 第一次启动很慢

便携 EXE 需要先解压约 400 MB 应用并接受系统安全扫描，首次等待约一分钟属于正常情况。后续启动通常更快。

### MissAV 或 123AV 的目标 RPS 很高，实际番号/秒却较低

RPS 是发送目标，真实速度还取决于站点响应时间、候选 URL 数量、解析标签、缓存命中、超时和限流。以“实际番号/秒”和 ETA 为准，不要只看并发数字。

### 出现 HTTP 429 或 Cloudflare Error 1015

这是站点限流，不是本地数据库损坏。先让当前冷却完成；123AV 收藏会固定单路并在异常后休息 10 秒。若仍连续触发，降低该网站的 RPS 或稍后继续，不要同时开启多个同站执行器。

### Telegram Bot 发现不到群

确认机器人已加入群组、`/setprivacy` 已设为 `Disable` 或机器人已是管理员，并在修改隐私设置后重新加入群组。随后在每个群发送一条新消息，再点“发现群组”。

### 123AV 明明登录却显示未登录

先在所选执行环境中打开 123AV 并确认页面可见账号信息。Chrome 扩展方式需确认扩展已连接；APP 内执行器需在它自己的账号窗口登录，两者登录状态不共用。

### Raindrop 显示“远端已存在，将更新”

表示账号中已有相同 URL。同步会更新该书签的标题、Tags 和目标 Collection，而不是创建重复项。

### 程序停止后仍显示未完成批次

“停止”是安全暂停。已完成结果已保存，剩余记录等待下次手动继续；若确定不再需要，可删除整个批次。

## 数据库、备份与升级

- 正式数据以本机 SQLite 为准，不以 GitHub 提交、标签或 Release 判断。
- 升级前退出旧 EXE，创建数据库备份，再启动新 EXE。
- 更换 EXE 不会覆盖用户数据；恢复备份会替换当前数据库状态。
- 全库归零只清空业务表，不删除备份、外观设置或 Windows 安全存储中的账号凭据。
- 不要把 `data/`、`*.db`、`*.sqlite*`、`.tmp-userdata/`、日志、导出目录或 Token 提交到仓库。
- 数据库支持搜索、筛选、排序、分页、单条编辑、Windows 式多选、批量修改/删除、CSV/TSV 导出、数据体检、备份恢复、批次删除和位置迁移。

## 输出文件

导出时会创建 `YYYYMMDD_HHMM_missav_import/` 文件夹，常见文件包括：

## 关键规则

- MissAV URL 会优先解析，可从 `/cn/xxx`、`/dm89/cn/xxx`、`/dm96/cn/xxx` 等链接中提取番号。
- Telegram HTML 中的 `message14298`、`MESSAGE-13`、`moodyz` 等噪声会被过滤。
- `需要查找` tag 只应对应 `not_found` 状态；如果链接可访问，请在处理结果或“番号库”中修正状态。
- `network_error` 表示超时、403/429、访问验证或防爬限制：会保存到永久历史并允许下次重试，但不会进入本批 Raindrop 导出。
- 正常详情页加载验证码/挑战公共脚本不会再被误判；只有缺少番号与详情证据的实际拦截页才会进入 `network_error`。
- 默认“智能”速度模式保留完整候选查询；6 路及以上属于主地址简查，可能漏掉字幕版、泄漏版或镜像页。
- Electron 网络层使用其支持的自动重定向模式；若仍有网络问题，请在“番号库 -> 维护 -> 运行日志”复制诊断报告。
- `need_manual_check` 只用于页面成功返回且正文含目标番号、但详情页证据不足的情况；候选 URL 本身不再算命中证据。
- SQLite 的 `codes` 表及其标签关联是当前长期主数据；`bookmarks`、`bookmark_collections` 仅保留为旧版兼容数据。
- 批次开始前会先把过滤后的番号写入 `processing_runs` / `processing_run_items`；不保存原始 TG 全文。MissAV 结果逐条落盘；123AV 只读查询使用每 100 条或 30 秒检查点，停止、完成和退出时强制落盘。
- 手动停止或程序异常退出不会把剩余项算作失败；下次启动可从“站点”工作台或“番号库 -> 处理批次”继续。

- 123AV 出现登录或访问验证时，在本地 Chrome 中手动处理；扩展只读取可见 DOM，不读取密码、Cookie、Local Storage 或 Session Storage。
- 结果页先选择 MissAV 或 123AV 板块，再在本站环节内筛选；网络错误重跑只作用于当前查询板块，并保留另一支线状态。Raindrop 导出只显示在 MissAV 板块。
- 大批量修改前建议先在“备份恢复”里创建备份。
