# Telegram 群组来源中心设计

目标版本：`v0.1.26`

## 目标

同一套番号处理流程提供三个互补入口：

1. Telegram Bot API：默认增量入口，只需 BotFather Token，对明确选择的最多 5 个群组、超级群组或频道读取加入后的新消息。
2. Telegram 用户 API：高级入口，登录用户自己的账号，可对明确选择的最多 5 个群组进行增量读取和手动历史补查。
3. Telegram Desktop 官方导出：导入这些群组过去的 JSON、HTML、单个文件或完整目录，建立历史底库。

三种入口最终转换成统一消息记录，复用现有番号过滤、消息历史去重和批次创建逻辑。三种入口都只生成预览，不自动启动 MissAV、123AV 或 Raindrop。

## 增量通道选择

Bot API 不需要 `my.telegram.org`、手机号验证码或个人账号会话，最适合“历史用导出文件建立底库，以后只处理自建群增量”的当前工作方式。它没有任意读取既有群历史的接口，Bot 加入前的消息不会补发，尚未取走的更新最长保留约 24 小时。用户 API 可列出账号本来就能访问的群，并主动读取历史。因此：

- 过去历史：优先使用 Telegram Desktop 官方导出。
- 今后增量：默认使用 Bot API；电脑长期关闭时用导出文件补齐。
- 需要补读历史或编辑回看：使用用户 API。

## Bot API 技术路线

- 协议：Telegram HTTP Bot API；只调用 `getMe` 和只读的 `getUpdates`。
- 凭据：用户在官方 `@BotFather` 创建机器人并取得 Token；不需要 `api_id/api_hash`。
- 安全存储：Token 只在 Electron 主进程使用，经 Windows `safeStorage` 加密保存；渲染层连接后不会取回 Token。
- 群组可见性：默认隐私模式只能看到命令和相关消息。用户必须通过 `/setprivacy` 关闭隐私模式并重新加入群组，或把机器人设为管理员。
- 群组发现：Bot API 不能枚举所有群，只能从收到的更新中识别群。每个目标群先发送一条普通识别消息，再由 APP 发现。
- 白名单：每个机器人最多选择 5 个群组；识别消息 ID 作为首次基线，正常结果只处理基线之后的消息。
- 全局游标：`getUpdates` 使用机器人级 `update_id`，不是独立群游标。所有已选群一起同步，未选群更新会忽略并推进，避免同一全局队列被某个群卡住。
- 安全确认：每批最多取 100 条；先把所有已选群消息写入 SQLite，再保存本地全局 `update_id`，下一次请求才向 Telegram 确认此前更新。崩溃重放只会产生历史重复，不会漏过未落库消息。
- 停止：当前批写入并保存游标后停止，不中断事务。
- Webhook 冲突：若机器人已被其他服务设置 Webhook，`getUpdates` 返回冲突；APP 不擅自删除远端 Webhook，提示改用新机器人或先在原服务停用。
- 保留期：服务器未取更新最长保留约 24 小时；本地写入后永久保留。当前版本不提供后台常驻采集。

## API 技术路线

- 协议：Telegram MTProto 用户 API。
- Node 客户端：`teleproto`，纯 JavaScript，无额外原生 DLL。
- 授权参数：用户从 `my.telegram.org` 获取自己的 `api_id` 和 `api_hash`。
- 登录流程：二维码扫码为推荐入口，短时二维码约 30 秒自动刷新；手机号 → Telegram 验证码作为备用。两条入口都支持可选 2FA 密码。
- 自动连接：存在有效加密会话时，APP 启动后只建立授权连接，不主动读取群消息；进入来源页后才按需加载群组。
- 会话：`StringSession` 仅在 Electron 主进程中使用，经 Windows `safeStorage` 加密后保存。
- `api_hash` 与会话一并加密；手机号、验证码、2FA 密码不写数据库、不写日志。
- 二维码 token 只转换成当前窗口内的短时二维码图片，不写 SQLite、文件或日志；取消扫码通过中止信号终止后台授权。
- 群组发现：连接后读取群组、超级群组和频道列表，只向渲染层返回群名称、类型、公开用户名、群 ID、管理状态和最新消息位置，不暴露访问哈希。
- 白名单：每个账号最多选择 5 个群组；未选择的私聊和群组不会进入消息读取。
- 默认基线：首次保存来源时记录群组当时的最新消息 ID，同时作为初始检查点。正常同步只读取这个基线之后的消息，不回扫历史。
- 增量：每个群独立保存基线、已完整覆盖的最高消息 ID、跨页游标和本轮目标消息 ID。
- 编辑回看：正常增量可回看基线之后最近 0～200 条已处理消息，用于发现编辑后的说明文字，但永远不会越过首次绑定基线。
- 手动补查：用户填写“补查起始日期”时可以临时越过基线回读指定日期之后的消息；所有结果仍经过消息历史和番号历史查重。
- 跨页续读：若某群未读消息超过单次上限，保存该群当前向旧消息分页的游标和本轮目标。只有该段补齐后才推进最高检查点。
- 群组顺序：最多 5 个群顺序读取，避免并发建立多个主会话；任一群异常不会覆盖其他群的游标。
- 限流：遇到 `FLOOD_WAIT_X` 按服务器要求停止本轮，不自动高频重试。
- 若 Telegram 要求仅官方客户端可完成的 reCAPTCHA，界面明确暂停并引导改用官方导出入口，不尝试绕过验证。

## 导出文件技术路线

- 支持 Telegram Desktop 官方 JSON 和 HTML。
- 推荐用户在每个目标群中分别使用“导出聊天记录”，可一次选择最多 5 组导出文件或目录。
- 支持选择 `result.json`、单个或多个 `messages.html`，以及完整导出目录。
- 账号级 JSON 只读取群组、超级群组和频道容器，忽略普通私聊与 Saved Messages；单独聊天导出的顶层 `messages` 仍可识别。
- JSON 文本可能是字符串或带格式实体的数组，统一展开为纯文本和链接。
- HTML 以消息容器和消息 ID 为边界；无法识别消息容器时降级为单文件正文。
- 不加载或复制图片、视频、语音等媒体，只读取文本、标题、链接、文件名和说明。
- 导出与 API 共用消息指纹；同一消息先通过文件导入、以后又被 API 补查到时不会重复产出番号。

## 统一消息模型

```text
sourceType       bot_group | api_group | export_json | export_html
accountKey       Telegram Bot ID 或用户 ID；文件无法识别时为空
chatKey          群组/超级群组/频道的稳定 Peer ID；文件无法识别时使用稳定导出键
messageId        Telegram 消息 ID；无法识别时使用稳定文件内序号
messageDate      Unix 时间或 ISO 时间
editedAt         编辑时间（如有）
text             仅在本次解析内存中存在
links            消息内链接
contentHash      规范化文本、链接和日期的 SHA-256
codes            现有番号过滤器结果
```

## 本地持久化

- `telegram_sources`：账号、群组、是否选中、首次绑定基线、检查点、跨页游标、最近状态与错误。
- `telegram_message_refs`：消息 ID、日期、内容指纹和提取番号；不保存完整正文。
- `telegram_import_runs`：每次群组读取或文件导入的数量、新增、重复、更新、番号和错误统计。

强去重键优先使用“账号 ID + 群组 ID + 消息 ID”；文件缺少账号或群组 ID 时使用日期与内容指纹补充去重。内容编辑后更新同一引用并重新计算番号。

移除群组只把它从增量白名单停用，不删除已提取消息指纹、番号或同步游标；以后重新添加时沿用原检查点。

## 界面

侧边栏“来源”保持三个二级页面：

- `机器人增量`：Token 连接、BotFather 设置说明、消息驱动群发现、最多 5 个白名单、全局更新游标、同步全部来源和安全停止。
- `个人账号 API`：账号授权、群组加载与搜索、最多 5 个白名单、独立群状态、同步单群/全部群、停止、补查日期。
- 授权区优先显示“扫码登录”；手机号验证码折叠为备用入口。已有加密会话自动连接，手动连接按钮只用于异常恢复。
- `官方导出文件`：选择 JSON/HTML/目录并建立历史底库。
- `导入预览`：消息数、新增、重复、更新和最终番号列表。
- `导入历史`：来源群、时间、统计、错误和检查点。

预览确认后把番号放入现有“过滤后的番号”框，仍由用户手动创建批次和选择网站。

## 安全边界

- API 客户端仅在主进程运行，渲染层拿不到会话字符串、`api_hash` 或群组访问哈希。
- Bot Token 仅在主进程和 Windows 加密文件中存在；数据库、日志和状态 IPC 不返回明文。
- IPC 返回的消息正文只用于当前导入，日志只记录账号 ID、群组 ID、数量和消息 ID，不记录正文。
- 不调用发送、转发、删除、已读、置顶等写操作。
- 不读取普通私聊，也不读取未加入白名单的群组历史。
- “退出并清除会话”会断开客户端并删除加密会话文件；Telegram 官方客户端中仍可随时终止该登录会话。
- 正式账号登录必须由用户完成；自动测试只使用模拟客户端和临时数据库。

## 验收

1. 连接后只列出群组/超级群组/频道，不把普通私聊作为候选。
2. 第 6 个来源无法保存；移除来源不删除原检查点。
3. 首次绑定以当前最新消息为基线，第一次正常同步不回扫旧历史。
4. 群组出现新消息后只产出新增内容；两个群的游标互不覆盖。
5. 填写补查日期时可读取基线之前的内容，但重复导出历史不会再次产出番号。
6. 单群超过读取上限时保存独立继续游标；停止后继续不会跳过中间消息。
7. 同一份 JSON/HTML 连续导入两次，第二次新增消息为 0。
8. 日志和 SQLite 均不出现会话字符串、`api_hash`、验证码、2FA 密码或完整消息正文。
9. 二维码能在过期后自动更新；取消后后台授权停止，扫码成功后下次启动自动连接保存会话。
10. Bot Token 能加密保存和清除，日志与错误信息不能出现 Token 或带 Token 的 API URL。
11. 机器人只能发现实际发过消息的群；第 6 个群无法选择，首次识别消息不会进入番号结果。
12. 同一批 Bot 更新必须在消息落库后才推进全局 `update_id`；中途停止或异常重放不会漏消息或重复产出番号。
